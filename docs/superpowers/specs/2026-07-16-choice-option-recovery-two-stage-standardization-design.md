# 粘连选择题选项恢复与两阶段标准化设计

> 状态：已部署验收；Python/TypeScript 粘连选项恢复、导入题两阶段标准化、本地全量测试、服务器 Docker 部署、AI smoke、业务 smoke 和真实 OCR PDF 导入 smoke 已于 2026-07-16 完成验证。

## 目标

修复 OCR 已保留选项文字、但把 C/D 粘入 B 选项的选择题。对于可证明的连续标签链，系统只重建 `tasks` 的边界，不新增或改写题目文字；同一编辑会话中，首次标准化只返回本地结果，第二次才强制调用 AI。

## 已确认的问题

当前 `split_tasks_options` 把每个 `\\task` 当成一个选项。若 OCR 输出为：

```latex
\\begin{tasks}(2)
\\task A 的内容
\\task B 的内容 C C 的内容 D．D 的内容
\\end{tasks}
```

则系统只保存 A、B。标准化路径又把“至少两个结构化选项”视为完整，直接返回本地无改动结果，因此不会调用 AI。选择题保护只能防止 AI 删除已知选项，不能从已损坏的两项结构推断 C/D。

## 范围

包含：

不包含：

- 全局标准化任务的交互协议改造。

## 选项恢复规则

恢复器只处理选择题的 `\\begin{tasks} ... \\end{tasks}` 块，并且必须同时满足：

1. 已存在至少两个非空 `\\task` 项；
2. 某个 `\\task` 内容中出现从下一预期标签开始的连续标签链，例如 B 后连续出现 C、D；
3. 每个候选标签位于文本区域、公式分隔符之外，且有明确标点或与选项内容之间的明确边界（包括后接公式或题图）；选项内容可以是公式、题图或普通文本，但标签链本身必须连续且明确；
4. 恢复后标签严格连续、无重复，且每个拆出的内容非空。

恢复操作仅把已有文本切分成新的 `\\task`，保留原始顺序和内容。任何参与恢复的标签链不完整、位于数学公式或完整 Markdown 图片的 alt/path 内、可能是变量/点名、内容为空、存在空 `\\task` 或未闭合的数学分隔符时都保持原文，不推测新选项。数学分隔符是否转义按其前连续反斜杠数量的奇偶性判定。某个选项中无关的公式变量/标签不会阻止其后独立、完整且位于公式外的标签链；公式文本仍归属该前一选项。

Python 与 TypeScript 都使用同一组 fixture：截图中的 `B ... C ... D．...` 必须恢复为 A/B/C/D；“点 D”“$C$”等非选项文字不得被拆分。

## 两阶段标准化契约

### 第一次点击：本地阶段

- 前端发送 `forceAi=false`；
- Java 导入题编排层把该请求转换为 worker 内部 `executionMode=local`；后端执行确定性的粘连选项恢复、既有 LaTeX 安全修复和渲染校验，但不调用 LLM；
- 结果以 `(questionId, 原始输入哈希, stage=local)` 区分并保存在本地应用缓存，响应标明 `executionPath=local`、`modelInvoked=false`；
- 若本地候选有改动，前端按现有候选流程写入编辑草稿，仍由用户正常点击“保存”持久化；
- 用户手工修改 Markdown 或刷新到不同题目时，前端的两阶段状态重置。

### 第二次点击：强制 AI 阶段

- 同一题目、同一轮编辑中第二次点击发送 `forceAi=true`；
- Java 导入题编排层把该请求转换为 worker 内部 `executionMode=force-ai`；后端必须跳过本地候选短路、标准化响应缓存和 LLM 路由缓存，实际请求 AI；
- AI 输入可先经过同一确定性边界恢复，但本地结果不得被标记为 AI 结果；
- 模型真实返回时，继续执行现有选择题保护、渲染校验和候选审阅；
- 强制 AI 失败时返回不可应用的可审阅结果：`modelInvoked=true`、`fallbackUsed=false`、`applyBlocked=true`，Markdown 保持当前值，并给出明确失败原因；不得回退并伪装成第一次的本地候选。

前端在本地阶段完成后将按钮/提示切换为“强制 AI 标准化”，并在强制调用失败时展示失败原因而不覆盖编辑稿。

### 路由边界

导入题公开接口 `/api/import-tasks/{taskId}/questions/{questionId}/standardize/ai` 由 Java 控制器拥有；它不是 Python 旧 API 的透明代理。Java 在保留客户端 `forceAi` 兼容字段的同时，只对该导入题路径派生并发送 worker 内部执行模式：`local` 或 `force-ai`。普通 `/api/markdown/standardize/ai` 和其他既有 worker 调用继续使用 `ai`，不因 `forceAi=false` 被误降级为本地阶段。

## 数据流

```text
当前 Markdown
  -> Java 导入题编排（forceAi=false -> local；forceAi=true -> force-ai）
  -> tasks 粘连恢复（确定性、无内容生成）
  -> 首次：本地缓存候选 -> 编辑草稿
  -> 第二次：forceAi=true -> 绕过全部应用缓存 -> LLM
  -> 选择题保护 / 渲染校验 -> 候选或人工复核
  -> 用户保存 -> 重新解析为结构化 options
```

## 验收标准

1. 截图同构 fixture 首次标准化输出四个独立 `\\task`，A/B/C/D 内容逐字保留，且 `modelInvoked=false`。
2. 同一轮第二次请求实际调用模拟 LLM，`modelInvoked=true`，并不命中本地或 LLM 缓存。
3. 强制 AI 失败时不产生可应用候选、不覆盖 Markdown，也不标注为本地或 AI 成功。
4. Python/TypeScript 对有效恢复和反例的选项结果一致。
5. 既有选择题图片保护、标准化缓存和全量相关测试继续通过。
6. 服务器部署后，对同构题目的首次/第二次标准化 smoke 均通过，OCR 运行态健康不回退。
