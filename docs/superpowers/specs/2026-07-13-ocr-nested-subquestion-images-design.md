# OCR 结构整理、尾部噪声过滤与嵌套小问题图守恒设计

## 目标

在 OCR 首次生成题目结构时完成安全的结构整理，并解决以下问题：

1. 试卷标题、答案区标题等非题目内容被拼入最后一道题或最后一个小问。
2. 小问拥有的题图虽然能在题目预览中渲染，但没有被 Java 题图索引递归收录，导致任务题图库无法定位。
3. AI 标准化或解析回写时，小问题图及其 Markdown 引用可能被模型返回内容覆盖。
4. 编辑器没有独立的小问题图管理入口，用户无法明确地把任务题图库图片关联到指定小问。

第 37 题的两张图明确归属于第（3）小问。目标结构必须保持：父题 `images=[]`，第（3）小问 `images=[图1, 图2]`，不得提升为父题共享图。

## 数据归属模型

题目 JSON 仍以现有结构为事实来源，不新增父子图片复制：

```text
question.images                 父题独占或父题共享图
question.subQuestions[i].images 第 i 个小问独占图
```

Java 的 `java_import_question_images` 表是检索索引，不改变事实归属。每个索引条目的 `raw_json` 增加：

```json
{
  "ownerKind": "subQuestion",
  "ownerQuestionId": "import_question_xxx",
  "ownerId": "q_37_sub_3",
  "ownerLabel": "(3)",
  "ownerPath": "subQuestions[2]"
}
```

父题图片使用 `ownerKind=question`。同一物理图片按 URL、path、storageFileId 或 name 去重，并保留 `owners` 数组；题图库展示的是一个资源及其归属，不把资源复制回父题。

## OCR 结构整理

OCR 构建题目和小问时继续按源文本 offset 分配图片。图片落在哪个小问的证据区间，就保存在该小问 `images` 中。

`NON_QUESTION_TAIL_RE` 扩展对以下尾部文档区块的识别：

- `2018-2019学年……期末物理试` 这类被 OCR 拆断、下一行才出现“卷”的标题；
- 单行完整试卷标题；
- 参考答案、试题解析等答案区标题。

只从已识别题目片段的尾部截断，不在题目正文中做全局替换。截断位置之前的 Markdown 图片引用和图片 offset 保持不变。

## 题图库递归索引

Java 同步题目时，先删除该父题旧索引，再遍历：

1. 父题 `images`；
2. `subQuestions`/`children`；
3. 每个小问自己的 `images`。

更新题目、小问、AI 回写和追加父题图片后均重建整棵题图索引，避免只更新父题造成小问题图消失。

Python worker 的任务题图库也递归合并 `task.questions` 中的父题和小问题图，为 Java 暂不可用或直连 worker 的场景提供一致结果。

## 标准化守恒

标准化可以修改题干、答案、解析、题型和选项文本，但不能隐式修改图片所有权。规则如下：

- 模型未返回小问时，保留现有小问完整结构。
- 模型返回小问文本时，按 `id`、`label`、顺序匹配，只用模型文本丰富现有小问；图片和位置字段以当前题目为准。
- 候选应用前逐小问检查现有图片引用。缺少引用、资产集合变化或所有权变化时阻止自动应用，题目进入人工复核。
- Java AI 回写同样不允许非空模型 `images` 覆盖已有小问题图。

## 编辑器交互

导入题编辑器在每个小问内新增“小问题图”区域：

- 展示该小问自己的图片；
- 可从任务题图库选择图片；
- 新增图片时只向该小问 Markdown 补充 `![](图N)`；
- 删除图片时只清理该小问题干、答案、解析和选项中的对应引用；
- 不改父题或其他小问。

任务题图库卡片显示“父题”或“小问 (3)”归属提示。前端图片去重优先使用 storageFileId、URL、path，避免索引 ID 和原题图片对象不同而重复显示。

## 验收条件

以第 37 题结构作为回归样例：

1. 父题没有图片。
2. 第（3）小问包含两张图片，源码包含两个稳定图片引用。
3. 尾部跨行试卷标题不在小问题干中。
4. Java 和 Python 任务题图库都能检索两张图，并标记归属于第（3）小问。
5. AI 标准化未返回图片或返回错误图片时，现有两张图及归属不变；删除引用的候选不能自动应用。
6. 编辑器只在第（3）小问显示和编辑这两张图。
7. Python 测试、Java 集成测试和前端 TypeScript/Vite 构建通过。
