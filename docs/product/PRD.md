# PRD：question-engine 题库加工能力与本地小平台

## 文档定位

本文件描述当前本期产品范围。项目不再按历史阶段口径拆分验收，而是统一定位为：

- `question-engine`：面向公司教育生态平台的题库加工能力服务。
- `local-platform`：本地可运行的小平台，用于演示、联调、验收和验证业务闭环。
- `backend`：唯一主后端，Java 负责业务 API、任务状态、数据表、文件存储和能力契约，Python worker 只保留 OCR、AI、LaTeX 和导出等必要执行能力。

历史文件名仍保留，是为了兼容已有文档链接，不代表项目仍按阶段拆分。

## 背景

教师和教研人员需要把试卷、答案解析和题图快速加工成可入库的标准题目数据。传统逐题录入会消耗大量人力，尤其在数学、物理等包含公式、图形和多问结构的试卷中，OCR 结果还需要人工校对和格式修复。

公司教育生态平台后续会统一管理用户、权限、组织、最终题库、审核流、知识点主数据和文件中心。本项目不应继续扩张成完整业务平台，而应提供“试卷到标准题目数据包”的加工发动机，并保留一个本地小平台验证发动机能力。

## 产品目标

本期目标是提供一套可本地运行、可平台集成、边界清晰的题库加工能力：

1. 接收试卷文件和可选答案文件。
2. 执行 OCR-Flow，默认使用 MinerU；provider adapter 输出 `CanonicalOcrBundle` 后统一进入 Post Process，替换 OCR 引擎不改变题库后处理流程。
3. 从 OCR 文本中拆分题目、提取题图、识别题型、选项和小问 / 子题。
4. 对 Markdown + LaTeX 进行本地公式标准化、风险检测和 AI 语义修复。
5. 使用大模型补全答案、解析、知识点候选、难度候选和分值候选；当 OCR 文本中包含答案解析时，优先基于原文证据匹配，不把模型自行求解结果当作教师原答案。
6. 提供可嵌入人工校验工作台，支持大题题干、小问题干/答案/解析、题图、知识点、难度和分值的人工修正。
7. 输出 `question-package.v1` 标准题目包，供平台最终入库。
8. 支持本地题库中心、知识点库和组卷中心，验证从导入到组卷导出的业务闭环。
9. 支持 Markdown、DOCX、PDF 导出：DOCX 优先使用 Markdown + Pandoc 链路，PDF 优先使用 XeLaTeX 预览模板，稳定处理数学公式、小问和作答空间。
10. 提供能力目录、静态 OpenAPI 契约、生成型 SDK 交付入口和 callback-flow，逐步满足公司平台集成要求。

## 用户与调用方

- 教师或教研人员：在本地小平台中上传试卷、校验题目、维护题库并组卷。
- 平台应用开发者：通过 OpenAPI/SDK 调用题目加工能力，把结果写入公司平台自有题库。
- 平台后端服务：调用 `question-engine` 创建加工任务、查询状态、获取标准题目包、接收回调。
- 项目开发和交付人员：使用本地小平台验证能力是否能独立运行和集成。

## 当前业务闭环

### 本地小平台流程

1. 用户进入 `local-platform`。
2. 在题目导入页创建导入任务，填写学段、学科、年级、地区、年份和标题。
3. 上传试卷文件，可选上传答案文件。
4. Java 后端保存原文件副本和文件元数据，再调用 Python worker 创建 OCR job。
5. 用户进入导入 OCR 工作台，左侧预览试卷或答案原文件，右侧查看 OCR 状态和待校验题目。
6. 系统生成题目、题图、选项、答案、解析、知识点候选、难度和分值。
7. 用户逐题校验 Markdown + LaTeX、题图、答案、解析和元数据；含小问的大题可逐小问维护答案、解析、分值和知识点。
8. 用户可点击 AI 标准化修复题干公式，或点击 AI 解析生成解析草稿并自动回填模型返回的答案。
9. 用户保存校验结果，单题或批量入库到本地题库。
10. 用户在题库中心检索、编辑、删除题目。
11. 用户维护知识点库。
12. 用户在组卷中心选题；含小问的大题可以只纳入部分小问，选择结果保存在试卷 `subSelections`，再编辑试卷、发布前预览，并导出 DOCX/PDF。

### 平台集成流程

1. 平台服务通过能力 API 创建题目加工任务。
2. 平台上传或传入试卷/答案文件。
3. `question-engine` 管理任务状态、OCR/AI/export job、文件元数据和失败原因。
4. Python worker 只执行 OCR、AI、LaTeX 修复和 DOCX/PDF 导出渲染。
5. 平台轮询任务或接收 callback-flow 回调。
6. 平台获取 `question-package.v1`。
7. 平台按自有权限、审核流、知识点主数据和题库表结构完成最终入库。

## 本期范围

### question-engine 能力

- `/api/engine`：四个核心模块能力目录，包括题目导入、题库能力、组卷能力和知识点能力。
- `/api/capabilities`：能力总目录，包括 `ocr-flow`、`question-processing`、`review-workbench`、`ai-flow`、`export-flow`、`file-flow`、`callback-flow` 和 `sdk-openapi`。
- `/api/capabilities/question-processing`：题目加工任务创建、查询和标准题目包输出。
- `/api/capabilities/ocr-flow`：OCR provider 合约和运行时状态。
- `/api/capabilities/ai-flow`：AI 标准化、AI 解析和答案解析匹配 job 编排。
- `/api/capabilities/export-flow`：导出 job 编排和文件存储。
- `/api/capabilities/file-flow`：原文件、题图、OCR 产物和导出文件的存储协议。
- `/api/capabilities/callback-flow`：任务事件、HTTP 回调签名、事件记录和重试入口。
- `question-engine/sdk`：TypeScript/Java SDK 草案；后续主路径应由正式 OpenAPI 生成。

### 本地小平台能力

- 题目导入：任务创建、历史任务管理、原文件预览、OCR 状态、人工校验和入库。
- 题库中心：题目搜索、筛选、查看、新增、编辑、删除、批量删除和题图管理。
- 知识点库：知识点新增、编辑、删除、批量删除和筛选。
- 组卷中心：试卷列表、选题、追加题目、排序、赋分、卷头、发布前预览和导出。

### 后端职责

- Java 是唯一主后端，默认端口 `8018`。
- Java 管理业务 API、任务元数据、状态机、数据表、文件存储、题图、导出任务、AI job、callback event 和平台能力契约。
- Java 默认使用 H2 本地库，企业部署可切换 MySQL。
- Java 文件存储默认写入本地目录，企业部署可切换 MinIO 或平台文件中心。
- Python worker 默认端口 `8000`，只保留 OCR provider、AI 模型调用、公式处理、DOCX Pandoc 导出和 PDF XeLaTeX 渲染。
- 旧 `/api/*` Python 业务接口只作为兼容迁移保留，不再作为新增业务入口。

## 不属于本项目主责

- 用户账号、角色、学校、组织和权限体系。
- 平台最终题库表结构、题目版本、审核流和发布治理。
- 平台知识点主数据治理。
- 平台长期文件归档和统一文件中心。
- 平台内其它业务应用的页面入口。
- 真实生产网关、租户隔离和统一认证。

## 成功标准

- 本地能一键启动 Java 后端、Python worker 和 `local-platform`。
- 用户能完成“上传试卷/答案 -> OCR -> AI 补全 -> 人工校验 -> 入库 -> 组卷 -> 导出”的闭环。
- Java 后端能独立暴露能力目录和题目加工能力 API。
- Python worker 可以被替换为其它 OCR/AI provider，而不影响 Java 能力契约。
- `question-package.v1` 可以作为平台最终入库的稳定输入。
- 本地小平台代码和 question-engine 交付代码边界清楚，不把演示页面误交付为平台能力。
